\section{프로그램 합성과 증명나무 합성} %공격 성공 보장도 여기서 설명
\begin{frame}
  \frametitle{프로그램 합성}
  \begin{itemize}
    \item<1-> 주어진 성질을 만족하는 프로그램을 자동으로 생성!
    \item<2-> ``입력이 1, 2, 3, 4일 때 각각 1, 3, 6, 10이 나오는 프로그램을 찾아''
    \item<3-> ``입력이 $x$이고 출력이 $y$일 때, $y^2 \le x < (y+1)^2$가 되는 프로그램을 찾아''
  \end{itemize}
\end{frame}

\begin{frame}
  \frametitle{프로그램 합성}
  \begin{itemize}
    \item<1-> 정답이 있는 문제의 경우
    \item<2-> bottom-up enumeration을 통해 유한 시간 안에 끝남을 보장
    \item<3-> 모든 프로그램을 빠짐없이 검사 $\rightarrow$ 여러 가지 트릭으로 시간 단축
  \end{itemize}
\end{frame}

\begin{frame}
  \frametitle{분석기 골탕먹이기 문제}
  \begin{itemize}
    \item<1-> ``분석기가 false alarm을 내는 프로그램을 찾아''
    \item<2-> Rice's Theorem 덕분에 정답이 있음을 보장
    \item<3-> 그렇다면 전탐색으로 쉽게 유한 시간 안에 풀 수 있겠네?
    \item<4-> \textcolor{red}{No.} false alarm이 진짜 나는지 알 방법이 없음
  \end{itemize}
\end{frame}

\begin{frame}
  \frametitle{증명나무 합성}
  \begin{itemize}
    \item<1-> 프로그램의 ``진짜'' 실행 의미를 알 수 없다는 게 문제
    \item<2-> 어떻게 하면 프로그램을 합성할 때 실행 의미까지 같이 끌고 다닐 수 있을까?
    \item<3-> 프로그램의 실행 의미를 Big-Step으로 정의해서 그 증명나무를 합성해보자
  \end{itemize}
\end{frame}

\begin{frame}
  \frametitle{증명나무 합성으로 분석기 골탕먹이기 문제 해결}
  \begin{itemize}
    \item<1-> 증명나무가 존재하는 프로그램은 항상 유한하게 종료됨
    \item<2-> 또한 최종 environment는 singleton set이 됨
    \item<3-> 따라서 분석기가 singleton set보다 큰 결과를 낸다면, false alarm이 발생함
    \item<4-> 역시 Rice's Theorem 덕분에 유한 시간 안에 끝남이 보장됨
  \end{itemize}
\end{frame}

\begin{frame}
  \frametitle{증명나무의 크기}
  \uncover<1-> {bottom-up enumeration은 크기 순서로 진행됨}

  \uncover<2-> {`크기'를 정의하는 것이 중요}

  \uncover<3-> {크기함수의 조건?}

  \begin{itemize}
    \item<4-> 모든 증명나무는 유한한 크기를 가짐
    \item<5-> 부분 증명나무의 크기를 더해 전체 증명나무의 크기를 구할 수 있으면 좋음
    \item<6-> \textcolor{red}{한 크기에 대해, 해당하는 증명나무의 수는 유한해야 함}
  \end{itemize}

    \item<3-> 모든 증명나무는 유한한 크기를 가짐? OK
    \item<4-> 부분 증명나무의 크기를 더해 전체 증명나무의 크기를 구할 수 있으면 좋음? OK
    \item<5-> 한 크기에 대해, 해당하는 증명나무의 수는 유한해야 함? OK
  \end{itemize}

  \only<6->{실행되지 않는 코드 조각을 위해 일반 프로그램 또한 같이 합성해야 함}
\end{frame}
 

\begin{frame}
  \frametitle{pruning 방법}
  \begin{itemize}
    \item<1-> 프로그램 합성에서는 실행 의미가 같은 프로그램은 하나만 남기고 버려도 됨
    \item<2-> 증명나무 합성에서는 실행 의미가 같더라도 막 제거하면 안 됨. 분석기가 분석을 다르게 할 수 있기 때문
    \item<3-> 그렇지만, 분석기를 `존중'하는 pruning은 가능
    \item<4-> $((p; q); r) = (p; (q; r)), (x + y) = (y + x)$ 와 같이 실행해 보지 않고 항상 미리 알 수 있는 경우
  \end{itemize}
\end{frame}
